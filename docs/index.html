Meet the Creature

Deep inside a quiet laboratory, a creature was born. It is unlike anything else on Earth: it can clone itself, endlessly and perfectly, on demand. There is only one supreme version of it — the original, the source of truth from which every copy descends.

Its purpose is simple. It grows fruit for humans. Fruit nourishes, delights, and sometimes heals — though every now and then, a piece carries an unexpected side effect.

The scientists' job is to coax better fruit from the creature, one careful experiment at a time.

The Lab = Your Repository

The lab is where the creature lives. Its walls hold every record, every fruit ever picked, every experiment ever logged. In Git, the lab is your repository.

You can build a brand new lab from scratch, or you can carry an existing one home with you to work on it.

git init                    # build a new lab
git clone <url>             # take an existing lab home

Either way, you now have a place where the creature — and its history — can live.

3 / 16

Cloning the Creature = Branches

No sane scientist pokes the supreme creature directly. If something goes wrong, the source of truth is ruined. Instead, they spin up a clone — an identical copy they can safely experiment on.

In Git, a clone of the creature is called a branch.

git branch feature-x        # create a clone named feature-x
git checkout -b feature-x   # create it AND step into the lab room with it

Whatever happens to feature-x stays with feature-x — until the scientist decides it's ready.

The Fruit = Commits

Every clone produces fruit. Each piece is a commit — a single, recorded change to the creature. Some fruit is sweet: a clean new feature, a long-awaited fix. Some fruit has side effects: the bugs nobody asked for.

Picking fruit is a two-step ritual. First the scientist places it on the tray for inspection, then writes it into the lab log.

git add fruit.txt           # place the fruit on the tray
git commit -m "Grow sweeter fruit"  # record it in the lab log

Many Scientists, Many Clones

The lab is busy. One scientist is breeding a clone for sweeter fruit, another is hunting down a side effect, a third is trying something wild. None of them disturb the supreme creature, and none of them disturb each other.

  • Isolation — experiments can fail without harming the source of truth.
  • Parallelism — many scientists work on many clones at once.
  • Focus — each clone has a single, clear purpose.
  • Reversibility — a bad clone can simply be discarded.

The Solo Scientist Workflow

Even alone in the lab, a scientist keeps a journal of every fruit — so missteps are reversible, experiments are isolated, and progress is never lost to a bad afternoon.

git init                              # build your lab
git checkout -b feature/sweet-fruit   # spawn a clone to experiment on
# ... edit files, run tests ...
git add .                             # gather the fruit from this clone
git commit -m "Grow first batch"      # record it in the journal
git checkout main                     # return to the supreme specimen
git merge feature/sweet-fruit         # bring the harvest home
git log --oneline                     # review the journal
  • Undo mistakes safely: a bad mutation is one git checkout away from being forgotten.
  • Try wild ideas without fear: spawn a clone, experiment freely, discard it if the fruit turns sour.
  • Document your progress: every commit is a dated lab note explaining what you grew and why.

The Merge Ritual

When a clone's fruit looks promising, the lab council convenes. They inspect each piece — taste, color, side effects — before allowing those improvements to flow back into the supreme creature. Only approved fruit joins the canonical genome.

git checkout main
git pull
git merge feature/sweet-fruit
git push
  • A pull request is a formal petition to graft a clone's fruit onto the supreme creature.
  • Peer scientists review the changes, suggest refinements, and flag risky mutations.
  • Nothing reaches the supreme creature until the council approves the merge.

When Fruits Collide

Two scientists tampered with the same gene of the supreme creature. The lab's machinery cannot reconcile the rival edits on its own — a human must step in and decide which trait wins, or how to weave them together.

<<<<<<< HEAD
flavor = "sweet"
=======
flavor = "tart"
>>>>>>> feature/tart-fruit
  1. Open the file and locate the conflict markers.
  2. Choose the correct trait — or combine them into something new.
  3. Run git add on the file, then git commit to seal the resolution.

The Lab Notebook

Every fruit ever grown is recorded in the lab notebook — who cultivated it, when it ripened, and why. That notebook is git log, the unbroken memory of the experiment.

git log                       # full history
git log --oneline             # one line per commit
git log --graph --all         # see all branches as a tree
git blame <file>              # who grew this fruit?
  • Trace a side effect back to the exact fruit that introduced it.
  • Understand why past scientists made the decisions they did.
  • Recover work you thought was lost in the chaos of the lab.

Rotten Fruit

Sometimes a fruit poisons the whole batch. The lab keeps two antidotes on the shelf — one safe, one dangerous. Never rewrite history that other scientists have already pulled into their clones.

git revert <commit>        # safe: grows a new fruit that undoes the bad one
git reset --hard <commit>  # dangerous: erases fruit from history
  • Prefer git revert on any branch shared with other scientists.
  • Reserve git reset --hard for your own local clone, before anyone has seen the fruit.
  • Rewriting shared history corrupts every clone downstream — the lab will not forgive it.

The Team Workflow

Labs are scattered across the world, each with its own scientists and clones. The central lab — origin — is the shared sanctuary where everyone syncs their findings so no discovery is lost.

git remote add origin <url>     # link to the central lab
git fetch origin                # see what's new (don't apply)
git pull origin main            # see what's new and apply
git push origin feature-branch  # share your clone
  • Pull before you push — absorb the central lab's progress before sending your own.
  • Push your branch often so peers can witness your clone's growth.
  • Never force-push to main — the supreme creature's lineage must never be rewritten.
12 / 16

Lab Etiquette

Good science is good housekeeping. A few rules keep the lab — and the supreme creature — healthy.

  • Grow small, focused fruit — each commit should do one thing, so it can be understood, reviewed, and undone on its own.
  • Label your fruit clearly — write descriptive commit messages in the imperative mood ("Add login flow", not "Added login flow").
  • Never experiment on the supreme creature — don't commit straight to main; spawn a branch and do your work there.
  • Sync before you share — git pull before you git push so your clone isn't stale when others receive it.
  • Review every fruit before merging — open a pull request and let a peer scientist inspect the work before it joins the supreme.

Scientist's Cheat Sheet

# Setup
git init                     # build a new lab
git clone <url>              # take an existing lab home

# Daily rhythm
git status                   # what's on the bench?
git add <file>               # gather a fruit
git commit -m "message"      # record it
git push                     # share with the central lab
git pull                     # bring in new fruit

# Branching
git branch                   # list clones
git checkout -b <name>       # spawn a new clone
git merge <name>             # fold a clone back into the supreme

# Rescue
git log --oneline            # read the notebook
git revert <commit>          # safely undo a fruit

Sharing Your Clone with the Team

Once your clone's fruit looks promising, send it to the central lab so the council can review it before it joins the supreme creature.

  1. Push your clone — git push -u origin feature/sweet-fruit uploads your branch and links it to the remote.
  2. Open the petition — gh pr create --fill (or use the GitHub web UI) opens a pull request describing what your fruit does.
  3. Respond to feedback — reviewers leave notes; commit fixes and git push again. The PR updates automatically.
  4. Merge when approved — gh pr merge --squash grafts your fruit onto the supreme creature on the central lab.
  5. Tidy the bench — git checkout main, git pull, then git branch -d feature/sweet-fruit to retire the spent clone.

The Supreme Cycle Continues

The supreme creature lives on through the careful work of many scientists, each tending their own clone, each fruit recorded in the notebook. Git is just the protocol for that ritual — the shared language that lets the lab keep growing without losing its history. Solo or with a team, the rhythm is the same: clone, grow, record, merge.

Now go grow something.
